# Visitor с примесями

### Проблема
---
Основная проблема: расширение функциональности сущностей в программе как в ФП, так и в ООП

В ФП это проявляется как дублирование функций, в ООП проявляется это как бесконечное наследование.
То есть в ООП можно без проблем добавить новый тип, но добавление нового метода вызывает затруднение, в то время как в ФП - добавление новой функции, практически ничего не стоит, в то время как при добавлении нового типа - придется возможно переопределить уже написанные функции.

Также корнем проблемы является безопасность типов при расширении класса. Когда у нас возникает поторебность в вызове определенного специфичного метода, который мы определили в наследнике, нам необходимо проводить downcast, что небезопасно.

В ФП также выражена эта проблема, только с другой стороны, например она возникает при обходе дерева или списка, которые хорошо формируют итоговые значения листьев. Но вдруг нам понадобится фильтрация листьев, или нахождение суммы до определенного значения, тогда нам необходимо использовать [[Функции высшего порядка]], которые определяют предикат при обходе какой-либо структуры.
### Решения
---
Решением данной проблемы является использование нескольких подходов и паттернов, но с некоторыми оговорками.

#### multiple dispatch

Для решения таких проблем в ФП - существует [[multiple dispatch|multiple dispatch]], который помогает добавить новую функцию лишь заменяя ее семантику, тем самым нужная функция будет выбрана исходя из типов аргументов. В ООП такой подход достаточно примитивен и определяется по одному аргументу - тип, который вызывает этот метод.

#### Visitor pattern

Данный паттерн используется как раз в ООП, как прослойка для добавления новых методов, пример:

дана такая иерархия классов
```
class Велосипед:
	def ТаймТриал(self): ...
	def БанниХоп(self): ...

class Шоссейник(Велосипед):
class Маунтинбайк(Велосипед):
```

нам необходимо определить разные специфичные методы для двух классов наследников

для этого мы как раз используем паттерн [[Паттерн Visitor|Visitor]]
```
interface IВелосипедVisitor:
	def ТаймТриал(м : Маунтинбайк): ...
	def БанниХоп(ш : Шоссейник): ...

class ВелосипедVisitor(IВелосипедVisitor):
	def ТаймТриал(м : Маунтинбайк): ...
	def БанниХоп(ш : Шоссейник): ...
```

теперь нам необходимо сделать следующее в наших классах
```
class Велосипед:
	def ТаймТриал(self): ...
	def БанниХоп(self): ...

class Шоссейник(Велосипед):
	def operate(v : IВелосипедVisitor):
		v.БанниХоп(self)
		
class Маунтинбайк(Велосипед):
	def operate(v : IВелосипедVisitor):
			v.ТаймТриал(self)
```

НО проблема заключается в том, что при если у нас планируется добавление нового функционала при закрытых классах, нам все больше и больше необходимо дополнять классы операциями: `operate1, operate2, ..., operateN`, что нечитабельно, непонятно, так и приходится вносить несколько изменений.
То есть Visitor сам по себе можно применить в каких-то конкретных случаях, но не везде.

Далее существуют несколько способов разрешения данной проблемы:

#### type classes

[[type classes|type classes]] - позволяет внедрять новые функции через класс-интерфейс с обобщенным типом, не меняя код класса. Выглядит примерно вот так:

```
class Printable<A>:
	def print(self, target : A): ...

class PrintableInt<int>(Printable<A>):
	def print(self, target : int):
		print(f'its int -> {target}')
	
class PrintableString<str>(Printable<A>):
	def print(self, target : str):
		print(f'its string -> {target}')

class Shower<A>:
	def show(value A, printable : Printable<A>):
		printable.print(value)
```

Таким образом, расширяемый класс лишь служит аргументом в соответствующей функции.

#### mix-in

Данный принцип помогает расширять класс засчет встраивания нужных частей, которые мы реализовали.

В Python делается с помощью множественного наследования (хочется уточнить что сам mix-in класс, который добавляется, он не должен инициализироваться, он должен только давать доступ к новым методам):

[example.py](https://github.com/aaboyarchukov/hard_work/tree/main/mixins/example.py)
```python
class XMLParser:
	def parse_to_xml(self, content : list[bytes]) -> XMLFile:
		...
		
class JSONParser:
	def parse_to_json(self, content : list[bytes]) -> JSONFile:
		...
		
class WebClient(XMLParser, JSONParser):
	def __init__(self, content):
		self.content = content
		
client = new WebClient(list[bytes][0, 1, 0, 1, 1, 1])
xml = client.parse_to_xml(client.content)
json = client.parse_to_json(client.content)
```

В Go это делается через embeeding структур:

[example.go](https://github.com/aaboyarchukov/hard_work/tree/main/mixins/example.go)

	Read(filePath string) []byte
}

type Writer interface {

type User struct {
